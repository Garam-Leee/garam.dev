# 코드를 덜 쓰는 개발자가 더 좋은 개발자일 수도 있다

개발을 처음 시작했을 때는 코드를 많이 작성하는 사람이 잘하는 개발자라고 생각했다.

새로운 기능을 빠르게 만들고,

복잡한 로직을 구현하고,

하루 종일 코드를 작성하는 모습이 멋있어 보였다.

그래서 나도 어떻게든 더 많은 코드를 작성하려고 했다.

그런데 몇 년 동안 실무를 하면서 생각이 조금 달라졌다.

오히려 좋은 개발자일수록 코드를 덜 쓰려고 한다는 것을 느끼게 됐다.

## 가장 좋은 코드는 없는 코드다

예전에 꽤 복잡한 기능을 구현해야 했던 적이 있었다.

처음에는 새로운 상태 관리 로직을 만들고,

공통 훅을 만들고,

여러 컴포넌트를 수정해야 한다고 생각했다.

하지만 요구사항을 다시 읽어보니 생각보다 단순한 문제였다.

기존 컴포넌트를 조금 확장하면 해결할 수 있었다.

결과적으로 수백 줄의 코드를 작성할 계획이었지만 실제 수정은 몇 줄로 끝났다.

당시에는 조금 허무하기도 했다.

하지만 시간이 지나고 보니 유지보수해야 할 코드가 적다는 것이 훨씬 큰 장점이었다.

## 기능을 만드는 것과 문제를 해결하는 것은 다르다

개발자는 종종 코드 작성 자체에 집중하게 된다.

하지만 사용자가 원하는 것은 코드가 아니다.

사용자가 원하는 것은 문제 해결이다.

예를 들어

> 사용자가 파일을 쉽게 다운로드할 수 있어야 한다.

는 요구사항이 있다고 해보자.

개발자는 종종

```txt
파일 생성 시스템 구축
다운로드 서버 구현
스토리지 설계
```

를 먼저 떠올린다.

하지만 실제로는 이미 존재하는 서비스를 활용하거나 브라우저 기능만으로 해결할 수도 있다.

결국 중요한 것은 얼마나 복잡하게 만들었는지가 아니라 얼마나 적절하게 해결했는지다.

## 내가 만든 코드보다 남이 만든 코드를 더 많이 읽는다

실무를 하면서 느낀 또 하나는

코드를 작성하는 시간보다 읽는 시간이 훨씬 많다는 점이다.

과거의 내가 작성한 코드도 결국은 남의 코드처럼 느껴진다.

6개월 전의 코드를 보면

```ts
const result = processData(data);
```

이 코드가 왜 필요한지 기억나지 않을 때가 있다.

그래서 요즘은 새로운 코드를 추가하는 것보다

기존 코드를 제거할 수 있는지를 먼저 고민하게 된다.

삭제할 수 있는 코드는 유지보수할 필요도 없기 때문이다.

## AI 시대가 되면서 더 크게 느낀 점

AI가 코드를 작성해주는 시대가 되면서 이런 생각은 더 강해졌다.

예전에는 구현 자체가 어려웠다.

지금은 AI가 대부분의 구현을 도와준다.

대신 더 중요한 질문이 남는다.

> 이 코드는 정말 필요한가?

AI는 500줄의 코드를 순식간에 작성할 수 있다.

하지만 필요 없는 500줄을 만드는 것도 순식간이다.

그래서 이제는

```txt
어떻게 구현할까?
```

보다

```txt
이걸 구현해야 할까?
```

를 먼저 고민하게 된다.

## 좋은 개발자는 복잡함을 줄인다

실무에서 존경했던 개발자들을 떠올려보면 공통점이 있었다.

코드를 많이 쓰지 않았다.

오히려 복잡한 문제를 단순하게 만들었다.

새로운 시스템을 추가하기보다 기존 시스템을 활용했고,

새로운 추상화를 만들기보다 필요 없는 추상화를 제거했다.

그래서 결과적으로 코드베이스가 더 단순해졌다.

당시에는 그 가치가 잘 보이지 않았지만 시간이 지나면서 차이가 커졌다.

## 마무리

예전에는 개발자의 생산성을

```txt
얼마나 많은 코드를 작성했는가
```

로 생각했다.

지금은 조금 다르게 생각한다.

```txt
얼마나 적은 코드로 문제를 해결했는가
```

가 더 중요한 기준인 것 같다.

물론 모든 코드를 줄일 수는 없다.

하지만 새로운 코드를 작성하기 전에

기존 것을 활용할 수는 없는지,

문제를 더 단순하게 정의할 수는 없는지,

정말 구현이 필요한 것인지,

한 번 더 고민하는 습관은 분명 도움이 됐다.

돌아보면 내가 가장 만족했던 프로젝트들은 가장 많은 코드를 작성했던 프로젝트가 아니라,

가장 적은 코드로 가장 큰 문제를 해결했던 프로젝트들이었다.
