# 좋은 코드는 읽기 쉬운 코드가 아니라 수정하기 쉬운 코드다

예전에는 좋은 코드를 읽기 쉬운 코드라고 생각했다.

변수명이 명확하고,
함수 길이가 짧고,
구조가 깔끔한 코드.

물론 지금도 중요하다고 생각한다.

하지만 실무에서 몇 번의 유지보수를 겪고 나니 조금 생각이 바뀌었다.

## 코드는 결국 바뀐다

새로운 요구사항이 추가되고,
기획이 변경되고,
예상하지 못한 예외 상황이 생긴다.

처음 작성한 코드 그대로 남아있는 경우는 거의 없었다.

그래서 중요한 것은

"지금 읽기 쉬운가?"

보다

"나중에 수정하기 쉬운가?"

였다.

## 수정하기 쉬운 코드

수정하기 쉬운 코드는 보통 역할이 명확했다.

어디를 바꿔야 하는지 알 수 있었고,
변경 영향 범위를 예측할 수 있었다.

반대로 읽기에는 깔끔해 보였지만 여러 책임이 섞여 있는 코드는 작은 수정도 어렵게 만들었다.

결국 좋은 코드의 기준은 작성 시점보다 변경 시점에 드러나는 것 같았다.

## 마무리

코드는 한 번 작성하고 끝나는 문서가 아니다.

시간이 지나면서 계속 수정된다.

그래서 요즘은 코드를 작성할 때

> "읽기 쉬운가?"

보다

> "3개월 뒤의 내가 수정하기 쉬울까?"

를 한 번 더 생각해보려고 한다.

생각보다 이 질문이 더 많은 문제를 발견하게 해준다.
