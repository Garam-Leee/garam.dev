## "그건 데이터팀 일 아닌가요?"

예전엔 저도 그렇게 생각했습니다. 개발자는 코드를, 데이터팀은 분석을 하는 거라고.

그런데 KPI 대시보드를 구축하면서 생각이 완전히 바뀌었습니다.

**데이터를 보지 않는 개발자는 자신이 만든 기능이 실제로 작동하는지 모릅니다.**

---

## GA4로 알 수 있는 것들

### 1. 내가 만든 버튼이 클릭되는가

```javascript
// 커스텀 이벤트 추적
gtag("event", "cta_click", {
  button_location: "hero",
  button_text: "시작하기",
  page_path: window.location.pathname,
});
```

이 코드 한 줄로 "히어로 섹션의 시작하기 버튼 클릭률"을 볼 수 있습니다. A/B 테스트의 기준이 됩니다.

### 2. 어디서 이탈하는가

퍼널 분석으로 회원가입 → 결제 흐름에서 어느 단계에서 이탈이 많은지 파악했습니다. 결과: 결제 폼의 카드 번호 입력 단계에서 43% 이탈.

원인: 폼 에러 메시지가 너무 늦게 나타남 → 실시간 validation으로 수정 → 이탈률 28% 감소.

### 3. 실제 사용 패턴

설계할 때 "당연히 쓰겠지"라고 생각했던 기능이 실제 사용률 2%였습니다. 반대로 "보조 기능"으로 만든 게 60% 사용률이었죠. 데이터가 없었다면 계속 잘못된 방향으로 리소스를 투자했을 겁니다.

## Datadog으로 알 수 있는 것들

GA4가 "사용자 행동"이라면, Datadog은 "시스템 건강 상태"입니다.

### Real User Monitoring (RUM)

```javascript
import { datadogRum } from "@datadog/browser-rum";

datadogRum.init({
  applicationId: "YOUR_APP_ID",
  clientToken: "YOUR_CLIENT_TOKEN",
  site: "datadoghq.com",
  service: "my-web-app",
  trackInteractions: true,
  trackResources: true,
  trackLongTasks: true,
});
```

이걸 심으면 자동으로 수집됩니다:
- Core Web Vitals (LCP, FID, CLS)
- JS 에러 발생 위치와 빈도
- API 응답 시간
- 사용자별 세션 리플레이

### 실제로 잡은 버그

배포 후 특정 iOS Safari에서만 레이아웃이 깨진다는 제보를 받았습니다. Datadog RUM 세션 리플레이로 재현해서 1시간 만에 수정했습니다. 기존엔 재현 자체가 하루 걸렸습니다.

## 데이터 기반 의사결정 사례

### 사례: KPI 대시보드 버튼 위치 변경

운영팀이 "내보내기 버튼을 더 잘 보이게 해달라"고 요청했습니다.

데이터 없이 작업했다면: "어디에 놓을지" 감으로 결정했을 겁니다.

**GA4로 현재 클릭 히트맵 확인** → 내보내기 버튼 클릭률 0.3% → 상단으로 이동 + 크기 키움 → 클릭률 4.1%

이 수치가 있어야 다음 스프린트에서 우선순위를 정할 수 있습니다.

## 프론트엔드 개발자가 데이터를 봐야 하는 이유

1. **내가 만든 것의 효과를 증명할 수 있다** — 팀 내 신뢰도 상승
2. **더 나은 의사결정을 할 수 있다** — 감이 아닌 근거
3. **빠르게 버그를 잡을 수 있다** — RUM 세션 리플레이
4. **기획자·PM과 같은 언어로 대화할 수 있다** — 협업 효율

데이터를 보는 개발자가 되세요. 생각보다 어렵지 않고, 생각보다 훨씬 가치 있습니다.
