## 도입 전 우리 팀의 상황

같은 버튼 컴포넌트가 코드베이스에 7개 있었습니다.

```tsx
// components/Button.tsx
// components/ui/Button.tsx
// components/common/Btn.tsx
// pages/dashboard/Button.tsx
// ...
```

신규 입사자가 들어오면 항상 같은 질문을 했습니다. *"어떤 버튼 컴포넌트를 써야 하나요?"*

PR 리뷰 때마다 "이건 기존 컴포넌트 쓰면 안 되나요?" 라는 코멘트가 달렸고, 디자이너와 개발자 사이에 항상 "이 컴포넌트 맞아요?" 라는 확인 작업이 필요했습니다.

---

## Atomic Design이란

Brad Frost가 제안한 UI 설계 방법론입니다.

```
Atoms       → 더 이상 쪼갤 수 없는 단위 (Button, Input, Icon)
Molecules   → Atom의 조합 (SearchBar = Input + Button)
Organisms   → 독립적인 UI 단위 (Header = Logo + Nav + SearchBar)
Templates   → 레이아웃 구조
Pages       → 실제 데이터가 채워진 페이지
```

이론은 명확합니다. 문제는 **팀에 어떻게 정착시키느냐**였습니다.

---

## 실패한 시도들

### 실패 1: 처음부터 완벽하게 만들려 했다

처음엔 Storybook을 완벽하게 세팅하고, 모든 컴포넌트를 문서화하려 했습니다. 두 달 동안 시스템만 만들다가 실제 제품 개발이 멈췄습니다.

**교훈**: 완벽한 시스템보다 팀이 쓸 수 있는 시스템이 더 가치 있습니다.

### 실패 2: 분류 기준이 모호했다

"이건 Atom이에요, Molecule이에요?" 라는 논쟁이 계속 생겼습니다. 하나의 컴포넌트를 어디에 넣어야 할지 매번 토론했습니다.

**교훈**: 분류 기준을 팀 컨텍스트에 맞게 단순화해야 합니다.

---

## 실제로 정착된 방법

### 1. 3단계로 단순화

5단계 Atomic을 3단계로 줄였습니다.

```
components/
  ui/       ← 재사용 가능한 기본 컴포넌트 (Button, Input, Card)
  blocks/   ← 비즈니스 로직 없는 조합 (UserCard, StatWidget)
  features/ ← 비즈니스 로직 포함 (LoginForm, DashboardHeader)
```

### 2. 새 컴포넌트 만들기 전 체크리스트

```
[ ] ui/ 에 비슷한 게 있는가?
[ ] props로 변형 가능한가?
[ ] 재사용 가능성이 있는가?
```

이 체크리스트 하나로 중복 컴포넌트가 80% 줄었습니다.

### 3. Storybook은 나중에

컴포넌트를 먼저 만들고, 안정화된 뒤에 Storybook에 추가했습니다. "Storybook 먼저" 방식은 오버헤드가 너무 컸습니다.

## 디자이너와 협업에서 가장 중요했던 것

컴포넌트 이름을 디자인 파일과 일치시키는 것이었습니다.

Figma에서 `Primary Button`이면 코드에서도 `<Button variant="primary">`. 이름 불일치가 가장 큰 커뮤니케이션 비용이었습니다.

## 마무리

Atomic Design은 방법론이지, 규칙이 아닙니다. 팀의 상황에 맞게 변형하고, 완벽함보다 실용성을 추구하세요. 처음엔 작게 시작해서 필요한 만큼 확장하는 것이 훨씬 효과적입니다.
